# sCrypt BSV Groth16 verifier (BN256) — provenance

sCrypt's **first** Groth16 verifier (Mihael Šinkec / MSinkec, "first zk-SNARK on
Bitcoin", Jul 23 2022), pulled from BSV **mainnet**. Over **BN256** (a.k.a.
alt_bn128 / BN254) — the **same curve** as the `groth16_contract` repo's
`BN256.cash` (prime
`21888242871839275222246405745257275088696311157297823662689037894645226208583`).
This is distinct from sCrypt's later **BLS12-381 testnet** line (see
`data/scrypt`, the `eba3` deploy), and it is the curve-matching single-tx BSV
reference for this suite.

## Transactions (WhatsOnChain, BSV mainnet, block 749,498, 2022-07-23)

- **Verifier (locking script)** — parent tx
  `320ba9fb3826c0bc66beed51edf2463e958b7274921563c5c90be62deabb725f`, output 0
  (the verifier lives in this ~11.7 MB tx's nonstandard vout[0] locking script;
  the tx's single input is a trivial P2PKH spend of funding tx
  `dc28b563bf7efe480327795ec109a820eba6e4b178d8b8358ed60f1e86369eb3`).
- **Proof (unlocking script)** — spending tx
  `24e8159c931ed1e64c8477c55d74dc6a4f8fe727888f8ad57ac2483db054bf24`, input 0
  (510 bytes; the constant-size Groth16 proof, spends the output above). Deployed
  and spent in the same block — a self-contained mainnet demo.

This is the ~11 MB "first iteration" the comparison doc cites; the "~1.5 MB
optimised" figure has no established mainnet txid (a writeup/testnet number).

## Why it fails on BCH beyond size: the trailing OP_RETURN

The verifier's arithmetic runs and the proof verifies (the loosened BCH 2026 VM
halts with a single `1` on the stack), but the locking script then ends in a
**reachable `OP_RETURN`**. It is the *only* OP_RETURN in the script, at byte
offset 11,717,401 of 11,717,408 — the very end — preceded by a run of `OP_NIP`
that collapses the stack to one item, then `OP_RETURN`, then 6 bytes of dead data
(`OP_PUSHBYTES_1 0x01 OP_0 OP_0 OP_0 OP_0`).

That is sCrypt's BSV pattern: post-Genesis BSV (active since Feb 2020, so it
applied to this Jul 2022 tx) treats `OP_RETURN` as a **terminator that succeeds
iff the stack holds a single non-zero item**. BCH treats an *executed* OP_RETURN
as an unconditional failure (`Program called an OP_RETURN operation.`). So this
verifier is BCH-incompatible for **two independent reasons**: (1) the ~11.7 MB
script and ~125x single-input op-cost budget, and (2) the BSV OP_RETURN
terminator semantics. The harness judges its correctness under the BSV rule
(`bsvOpReturnTerminator` in the implementation -> the `BSV OP_RETURN` tag in the
leaderboard) while keeping the BCH-compatibility verdict strict.

## Files

| file | what |
|------|------|
| `parent-tx.hex` | raw ~11.7 MB parent tx (contains the verifier locking script in vout[0]) |
| `spending-tx.hex` | raw 510 B spending tx (contains the proof unlocking script in vin[0]) |
| `groth16-locking.asm` | disassembled verifier opcodes (regenerated by extract) |
| `groth16-unlocking.asm` | disassembled proof/witness, push-only (regenerated by extract) |

The `.hex` and `.asm` blobs are gitignored. Re-fetch them from WhatsOnChain and
regenerate the listings:

```
pnpm fetch:scrypt-bn256    # downloads parent-tx.hex (~23 MB hex) + spending-tx.hex
pnpm scrypt-bn256:extract  # regenerates the groth16-*.asm opcode listings
pnpm scrypt-bn256:run      # replays the real mainnet spend on the loosened BCH 2026 VM
```
